OneDeck 物理卡片排列方案

目标排列描述：参考图展示的是一种「平滑曲线级联堆叠（Smooth Curve Cascade Stack）」。最前方卡片最大、位于右下；前段卡片沿平滑曲线朝「左上」方向展开，同时逐步缩小、间隔逐步减小；到达转折点后曲线自然地弯向「右上」，卡片尺寸趋于稳定，间隔收缩至最小值并保持。整体形成一条连续的弧线纵深，而非生硬折线。

本 Demo 使用 20 张卡片，复现参考图的配色与折线级联排列，并提供可调参数观察不同 X 值与间距曲线的效果。

方案A：绝对定位 + CSS Transform
方案B：容器堆叠 + Transform
方案C：Canvas 2D 绘制
方案D：WebGL/Three.js 透视

方案 A：绝对定位 + CSS Transform（Unity 对应：RectTransform 手动计算）

在 Unity 中相当于每张卡片的 RectTransform 通过脚本按索引计算 localPosition、localScale。逻辑简单、可控性最高，是最直接的实现方式。

实现成本

  • 低：只需在 CombatUXManager 中增加排列计算函数
  • 无需额外包或 Shader

优势

  • 完全可控，便于与现有 DOTween 动画集成
  • 易于响应 Reveal / Bury / Stage 等事件实时更新
  • 原生 UI 或 World Space 都适用

劣势

  • 卡片数量极大时（>100）布局计算会线性增长
  • 需要手动处理层级排序（Sorting Order）

方案 B：容器堆叠 + Transform（Unity 对应：父物体作为布局锚点）

将所有卡片放入一个 StackContainer，父物体只负责整体位置，子物体通过局部坐标偏移。Unity 中可保留现有 CardPhysObjScript 的父子结构，仅修改局部偏移量。

实现成本

  • 中低：需要重构 CardPhysObjScript 的父级关系
  • 保留现有组件，改动集中在布局函数

优势

  • 整体移动牌库时只需移动父物体
  • 动画可复用现有 Tween，对现有代码侵入性小
  • 便于做整体入场/离场动画

劣势

  • 卡片单独飞出（如 Attack / Exile）时需要临时脱离父级
  • 层级控制仍需要显式设置 sibling index 或 sorting order

方案 C：Canvas 2D 绘制（Unity 对应：自定义 IMGUI/Editor 或 SpriteRenderer 动态绘制）

不依赖单个 Card GameObject，而是在一个 RenderTexture 或 Canvas 上一次性绘制所有卡片。Unity 中可用 SpriteRenderer + 动态 Texture，或 UI RawImage + RenderTexture。

实现成本

  • 中高：需要实现绘制管线，交互检测较复杂
  • 在 Unity 中通常需要 RenderTexture 或自定义 Mesh

优势

  • 性能最好，大量卡片时 Draw Call 极低
  • 可轻松实现复杂几何、阴影、渐变
  • 适合用作远景牌库或缩略图

劣势

  • 单张卡片交互（Hover/Click/Drag）需要额外实现命中检测
  • 与现有 CardPhysObjScript 动画体系不兼容
  • 开发和调试成本高

方案 D：WebGL/Three.js 透视（Unity 对应：真正的 3D 透视相机）

将卡片作为 3D 空间中的 Quad/Plane，利用透视相机的自然深度产生缩放。Unity 中只需把卡片放在 World Space 3D 空间中，使用 Perspective Camera，Z 轴深度即可产生大小差异。

实现成本

  • 中：需要把 UI 卡片改为 World Space 3D Quad 或启用 3D 旋转
  • 相机和光照需要重新调整

优势

  • 透视效果最自然，无需手动缩放计算
  • 可叠加 3D 翻转、弯曲、光照等高级效果
  • 视觉上限最高

劣势

  • 与现有 2D UI 动画体系冲突大，改造成本高
  • 需要处理 3D 层级、遮挡和点击射线检测
  • 对美术资源一致性要求高
